# Focal — AI Agent Guide

Focal is a bidirectional sync tool between a personal GitHub Projects v2 Kanban
board and origin repo project boards. It lets a contributor manage all their
assigned issues from a single personal board instead of jumping between many
repo boards.

## Repo contents

| File/Dir | Role |
|---|---|
| `focal.py` | CLI entry point — `python3 focal.py board sync` / `python3 focal.py board setup` |
| `focal/` | Python package — all logic lives here |
| `focal/sync.py` | Sync logic (`Syncer` class) |
| `focal/wizard.py` | Interactive setup wizard |
| `focal/gh.py` | gh CLI subprocess wrapper — all GitHub API calls |
| `focal/config.py` | `Config` dataclass, load/save `config.json` |
| `focal/state.py` | State file load/save |
| `focal/log.py` | Logging setup (file + optional stdout) |
| `sync.sh` | Thin wrapper: `exec python3 focal.py board sync` |
| `setup.sh` | Thin wrapper: `exec python3 focal.py board setup` |
| `config.json` | **Gitignored** — user's personal config (generated by setup) |
| `config.example.json` | Template showing all config keys |
| `status_map.json` | **Gitignored** — auto-generated origin→personal status name mappings |
| `requirements.txt` | Python deps: `typer[all]` |
| `launchd/com.your-username.focal.plist` | macOS launchd agent template for hourly sync |

## Runtime data (not in repo)

| Path | Purpose |
|---|---|
| `~/.focal/state.json` | Tracks last-known status of every synced issue |
| `~/.focal/logs/YYYY-MM-DD.log` | Daily rotating log files |
| `~/.focal/logs/launchd.log` | stdout capture from the launchd agent |
| `~/.focal/logs/launchd-error.log` | stderr capture from the launchd agent |

## Onboarding a new user

When a user opens this repo for the first time, guide them through these steps
in order. Check each prerequisite before moving on.

### Step 1 — Verify prerequisites

```bash
gh auth status        # must show "Logged in" with repo + project scopes
gh --version          # 2.x or later
python3 --version     # 3.10 or later
```

If `gh` is not installed: `brew install gh` (macOS) or see https://cli.github.com.

If `gh` is installed but missing the `project` scope:
```bash
gh auth refresh -s project
```

Install Python dependencies:
```bash
pip3 install -r requirements.txt
```

### Step 2 — Create a personal GitHub Projects v2 board

If the user doesn't have one yet:
1. Go to https://github.com/users/YOUR_USERNAME/projects
2. Click **New project** → choose **Board** layout
3. Add a **Status** single-select field with these options (recommended):
   ```
   🆕 New  ·  📋 Backlog  ·  🔖 Ready  ·  🏗 In progress  ·  ✋ Blocked  ·  👀 In review  ·  ✅ Done
   ```
4. Note the project URL — you'll need the number from it (e.g. `projects/3`)

### Step 3 — Run the setup wizard

```bash
python3 focal.py board setup
# or: ./setup.sh
```

The wizard will prompt for:
- **Board URL** — e.g. `https://github.com/users/alice/projects/3`
- **GitHub username** — the assignee to filter issues by
- **Done status name** — the exact name of your Done column (e.g. `✅ Done`)
- **Repos to sync** — choose from three modes:
  - *Manual list* — type repos one by one (`owner/repo`)
  - *Interactive select* — browse and pick from accessible repos
  - *Full scan* — scans ALL accessible repos (slow, may take minutes)

It will then:
- Inspect Status columns in every origin project
- Report mismatches vs the personal board
- Offer to map incompatible status names via `status_map.json`
- Write `config.json` (gitignored)

### Step 4 — Run a manual sync to verify

```bash
python3 focal.py board sync
# or: ./sync.sh
```

Watch the output. You should see issues being added and statuses inherited.
The final line shows a summary:
```
[INFO ] Sync complete — added: N  inherited: N  pushed: 0  stale: 0
```

If you see `[WARN]` lines about a status name, an origin project has a status
that doesn't match any personal board option. Handled gracefully — skipped with
a warning. Fix by re-running `python3 focal.py board setup`.

### Step 5 — Schedule recurring sync

**macOS (launchd — recommended over cron):**

launchd survives sleep/wake cycles and starts on login.

```bash
cp launchd/com.your-username.focal.plist ~/Library/LaunchAgents/com.YOUR_USERNAME.focal.plist
```

Edit the copied plist — replace `YOUR_USERNAME` with your macOS username
(`whoami`) and `/path/to/focal` with the absolute path to this repo. Then:

```bash
launchctl load ~/Library/LaunchAgents/com.YOUR_USERNAME.focal.plist
launchctl list | grep focal   # should show the agent with exit code 0
```

To trigger an immediate run:
```bash
launchctl start com.YOUR_USERNAME.focal
```

**Linux / alternative (cron):**
```bash
(crontab -l 2>/dev/null; echo "0 * * * * $(pwd)/sync.sh") | crontab -
```

## Common tasks

### Add or remove repos

Edit `config.json` directly:
```json
"repos": [
  "owner/repo-one",
  "owner/repo-two"
]
```
Re-running `python3 focal.py board setup` overwrites `config.json`, so prefer direct
edits after initial setup.

### Reset sync state (re-baseline everything)

Delete the state file so the next run treats all board items as new and
re-inherits statuses from origin:
```bash
rm ~/.focal/state.json
python3 focal.py board sync
```

### Check logs

```bash
# Follow today's log live
tail -f ~/.focal/logs/$(date '+%Y-%m-%d').log

# See only warnings
grep 'WARN' ~/.focal/logs/*.log

# Check last sync summary
grep 'Sync complete' ~/.focal/logs/$(date '+%Y-%m-%d').log | tail -1
```

Log format:
```
[YYYY-MM-DD HH:MM:SS] [INFO  ] Board: #3 (PVT_kwH...)
[YYYY-MM-DD HH:MM:SS] [INFO  ] Adding: https://github.com/some-org/some-repo/issues/42
[YYYY-MM-DD HH:MM:SS] [WARNING] 'In Progress' not found in "some-org's project" — skipping
[YYYY-MM-DD HH:MM:SS] [INFO  ] Sync complete — added: 3  inherited: 3  pushed: 1  stale: 0
```

### Debug a specific issue

```bash
python3 focal.py board sync  # output goes to stdout when running interactively
# or enable verbose tracing:
PYTHONINSPECT=1 python3 focal.py board sync
```

### Check launchd agent status

```bash
launchctl list | grep focal   # shows PID and last exit code
# exit code 0 = last run succeeded
# exit code non-zero = last run failed — check:
cat ~/.focal/logs/launchd-error.log
```

## Architecture notes

- **CLI entry** — `focal.py` uses `typer`. Subcommands: `sync` and `setup`.
  `sync.sh` / `setup.sh` are thin one-line wrappers for backward compatibility
  and launchd.
- **`focal/gh.py`** — all GitHub API calls. Uses `subprocess` to call `gh` CLI.
  GraphQL for project mutations, REST (`gh issue list`) for issue listing. No
  direct API token needed beyond `gh auth`.
- **`focal/sync.py`** — `Syncer` class. Three actions per board item:
  `SET_DONE` (stale), `INHERIT_ORIGIN` (new item), `PUSH_TO_ORIGIN` (card moved).
- **State file** (`~/.focal/state.json`) — JSON map of
  `issue_url → {personal_status, origin_status}`. Detects diffs between runs.
- **Conflict resolution** — personal board wins. If both sides changed between
  syncs, the personal board status is pushed to origin.
- **Status matching** is emoji-normalized: `🏗 In progress` and `In progress`
  match. See `normalize()` in `focal/sync.py`.
- **Status map** (`status_map.json`) maps `{project_id: {origin_name: personal_name}}`.
  Used when an origin project uses fundamentally different status names.

## Key config fields (`config.json`)

| Field | Description |
|---|---|
| `board_owner` | GitHub username who owns the personal board |
| `board_number` | Project number from the board URL |
| `assignee` | GitHub username to filter assigned issues |
| `status_field_id` | Node ID of the Status field on the personal board |
| `done_status` | Exact name of the Done option (e.g. `✅ Done`) |
| `repos` | Array of `owner/repo` strings to sync |
| `state_file` | Path to the JSON state file (default: `~/.focal/state.json`) |
| `log_dir` | Directory for daily log files (default: `~/.focal/logs`) |

## Things to be careful about

- `config.json` and `status_map.json` are gitignored — never commit them.
- The `gh` token needs `project` scope. Without it, project queries silently
  return empty results and no issues will be synced.
- `gh project item-add` is idempotent — safe to call multiple times for the
  same issue URL. Duplicates are silently ignored.
- Status pushes to origin are best-effort. A missing status option logs a
  warning and moves on — it will not fail the whole sync.
- The sync is poll-based (no webhooks). GitHub does not emit events for
  personal project board card moves.
- Python 3.10+ required (uses `list[str]` and `dict[str, str]` type hints
  without importing from `typing`).
