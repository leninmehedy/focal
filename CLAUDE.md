# Focal — Claude Code Guide

This repo contains two shell scripts that sync a personal GitHub Projects v2
Kanban board bidirectionally with origin repo project boards.

## What's in this repo

| File | Role |
|---|---|
| `setup.sh` | Interactive wizard — checks tools, inspects projects, fixes mismatches, writes `config.sh` |
| `sync.sh` | Main sync script — pull new issues, push status changes, mark stale items Done |
| `config.sh` | **Gitignored** — user's personal config (generated by `setup.sh`) |
| `config.example.sh` | Template showing all available config keys |
| `status_map.json` | **Gitignored** — auto-generated name mappings for projects with different status labels |

## How to help a user get started

1. **Check prerequisites first:**
   ```bash
   gh auth status
   gh --version
   python3 --version
   ```

2. **Run setup** — this is always the first step for a new user:
   ```bash
   chmod +x setup.sh sync.sh
   ./setup.sh
   ```
   The wizard will prompt for: board URL, GitHub username, repos to track.
   It produces `config.sh` and optionally `status_map.json`.

3. **Run a manual sync** to verify everything works:
   ```bash
   ./sync.sh
   ```

4. **Schedule recurring sync** (hourly is a good default):

   **macOS (launchd — recommended):**
   ```bash
   cp launchd/com.your-username.Focal.plist ~/Library/LaunchAgents/
   # Edit the plist: replace YOUR_USERNAME and /path/to/Focal
   launchctl load ~/Library/LaunchAgents/com.your-username.Focal.plist
   ```

   **Linux / alternative (cron):**
   ```bash
   (crontab -l 2>/dev/null; echo "0 * * * * $(pwd)/sync.sh") | crontab -
   ```

## Common tasks

### Add or remove repos
Edit the `REPOS` array in `config.sh` directly. Re-running `./setup.sh` will
overwrite `config.sh`, so prefer manual edits after initial setup.

### Reset sync state
Delete the state file to re-baseline all item statuses from origin:
```bash
rm ~/.Focal/state.json
```
The next sync run will treat all board items as new and re-inherit origin statuses.

### Check logs
Logs go to `~/.Focal/logs/YYYY-MM-DD.log` (one file per day).
```bash
# Follow today's log
tail -f ~/.Focal/logs/$(date '+%Y-%m-%d').log

# Grep warnings across all days
grep '\[WARN\]' ~/.Focal/logs/*.log

# Check last sync summary
grep 'Sync complete' ~/.Focal/logs/$(date '+%Y-%m-%d').log | tail -1
```

**Log format:**
```
[YYYY-MM-DD HH:MM:SS] [LEVEL] message
```
Levels: `[INFO ]`, `[WARN ]`, `[ERROR]`. Every run ends with a summary line
showing counts: `added`, `inherited`, `pushed`, `stale`.

### Debug a specific issue
Run `sync.sh` with `bash -x` to trace execution:
```bash
bash -x ./sync.sh 2>&1 | head -100
```

### Fix Status column mismatches manually
If `setup.sh` couldn't fix a mismatch (e.g. no write access to an origin org
project), the sync will still work via `status_map.json`. To fix it properly,
re-run `./setup.sh` — it will re-inspect and offer fixes again.

## Architecture notes

- **State file** (`~/.Focal/state.json`) — a JSON map of
  `issue_url → {personal_status, origin_status}`. Tracks what was last seen so
  the script can detect diffs between runs.
- **Conflict resolution** — personal board wins. If both sides changed, the
  personal board status is pushed to origin.
- **Status matching** is emoji-normalized: `🏗 In progress` and `In progress`
  match. This is handled by the `normalize()` Python function in `sync.sh`.
- **Status map** (`status_map.json`) maps `{project_id: {origin_name: personal_name}}`.
  Used when an origin project uses fundamentally different status names (e.g.
  `Todo` instead of `🆕 New`).
- All GitHub API calls go through the `gh` CLI (GraphQL for project mutations,
  REST for issue listing). No separate API token is needed.

## Key config variables

| Variable | Description |
|---|---|
| `BOARD_OWNER` | GitHub username who owns the personal board |
| `BOARD_NUMBER` | Project number from the board URL |
| `ASSIGNEE` | GitHub username to filter assigned issues (usually same as `BOARD_OWNER`) |
| `STATUS_FIELD_ID` | Node ID of the Status field on the personal board |
| `DONE_STATUS` | Exact name of the Done option (e.g. `✅ Done`) |
| `REPOS` | Array of `owner/repo` strings to sync |
| `STATE_FILE` | Path to the JSON state file |
| `LOG_DIR` | Directory for daily log files (default: `~/.Focal/logs`) |

## Things to be careful about

- `config.sh` and `status_map.json` are gitignored — never commit them.
  They may contain project IDs and personal configuration.
- The `gh` token needs `project` scope to read/write GitHub Projects v2.
  Without it, project queries will silently return empty results.
- `gh project item-add` is idempotent — safe to call multiple times for the
  same issue URL.
- Status pushes to origin are best-effort. If an origin project doesn't have a
  matching status option, the script logs a warning and continues. It will not
  fail the whole sync.
- The sync is poll-based (no webhooks). GitHub does not emit events for
  personal project board card moves.
