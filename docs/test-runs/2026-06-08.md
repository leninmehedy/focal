## Failures & Issues

### BUG-1 · Board sync — Incremental sync false-stale ❌

**Test:** BS3  
**Severity:** High — breaks board sync for steady-state repos  
**Command:** `focal board sync` (second run and every run after)  
**Symptom:** After first full sync, every subsequent incremental sync marks all board items as `Stale (closed/unassigned)` even when the issues are still open and assigned.  
**Root cause:** Incremental sync queries GitHub for issues `updated_since` the last sync timestamp. Issues that haven't changed since last sync are not returned. The stale detector then sees the board item is NOT in the "open assigned" result set and incorrectly marks it stale. Only the first (full) sync correctly tracks all items.  
**Workaround:** Delete `~/.focal/state.json` to force a full sync each time (expensive).

---

### BUG-2 · Board sync — `done_status` mismatch with `--use-board` ⚠

**Test:** BS6  
**Severity:** Medium — sync can't move stale items to Done column  
**Symptom:** When attaching an existing board with `--use-board`, the wizard writes `done_status: "✅ Done"` (the canonical default) into config. If the board's actual Done column is named "Done" (no emoji), every stale-move attempt logs `WARNING: No personal board option for '✅ Done' — skipping`.  
**Root cause:** `--use-board` setup doesn't introspect the existing board's Status field options to match the real Done column name.  
**Workaround:** Manually edit `~/.focal/config.json` → set `"done_status": "Done"` to match the board.

---

### BUG-3 · PM init — Non-existent repo silently succeeds ❌

**Test:** PI10  
**Severity:** Medium  
**Command:** `focal pm init nonexistent/norepo`  
**Symptom:** When targeting a repo that doesn't exist, init logs `⚠ Could not create label 'epic': HTTP 404` and `⚠ Could not create label 'story': HTTP 404` but continues, creates local files, and exits 0 with "Init complete!".  
**Expected:** Detect 404 on first GitHub call, print clear error "Repository nonexistent/norepo not found", exit non-zero.

---

### BUG-4 · adopt-plan — Stories not created on first `--apply`; not idempotent ❌

**Tests:** AP2, AP3, AP4  
**Severity:** High — core workflow broken for new users  
**Symptom:** Running `focal pm adopt-plan owner/repo --apply` creates all epics from `plan.md` but creates **0 stories**. A second `--apply` run then creates the stories (which is not idempotent behavior).  
**Root cause:** Stories require their parent epic's `id` to be in local state before they can be linked. In the first pass, epics are created and written to GitHub but the in-memory state is not updated before the story-creation loop runs. On the second pass, epics are already in state so stories are processed correctly.  
**Workaround:** Run `focal pm adopt-plan owner/repo --apply` twice on a fresh repo.

---

### BUG-5 · Plan — `--goals` flag has no effect ❌

**Test:** PL3  
**Severity:** Low  
**Command:** `focal pm plan ... --goals "I1:Ship E3,I2:Close E1"`  
**Symptom:** Command accepts the flag and exits 0, but the generated `iteration-planning.md` contains no goal text for any iteration.

---

### BUG-6 · Velocity — Parser returns zeros; parses comment block ❌

**Tests:** V1, V2  
**Severity:** Medium  
**Command:** `focal pm velocity owner/repo`  
**Symptoms:**  
1. All SP values (Planned, Delivered, Carry-over, Capacity) show as 0 despite retro-log.md containing valid data (e.g. `Planned: **6 SP**`).  
2. Goal-met column shows `❌` for I1 even though `--goal-met` was passed (retro-log shows `**Met:** ✅ Yes`).  
3. Commented-out template block at the top of retro-log.md is parsed as a real iteration row, producing a phantom `I1` entry.  
**Root cause:** Velocity parser likely uses regex that doesn't match the bold markdown (`**6 SP**`) or the `✅ Yes` format. Comment block (`<!-- ... -->`) not being skipped.
